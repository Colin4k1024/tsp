# 需求澄清与范围管理演练

本文演示一个需求从原始诉求到可进入主链的完整澄清过程。目标是避免模糊需求直接进入执行阶段。

## 1. 场景

- 原始诉求：业务方希望“审批列表更好用”
- 问题：目标模糊，范围容易蔓延
- 目标：沉淀成可执行需求和可测验收标准

## 2. 推荐链路

1. `/team-intake`
2. `/team-plan`
3. `/handoff`

必要时在 intake 前先由产品经理完成业务澄清，再进入主链。

## 3. 关键输出

- 问题定义
- In Scope / Out of Scope
- 用户故事和验收标准
- 优先级和风险提醒

## 4. 合格结果的检查点

- 研发能看懂目标和边界
- Project Manager 能基于此拆依赖和里程碑
- QA 能据此写出测试点

## 5. 常见错误

- 用模糊形容词代替明确目标
- 只说想要什么，不说不做什么
- 验收标准没有行为或数据依据
